## Project Overview

BNETDocs Static is the GitHub Pages-hosted static version of bnetdocs.org, a documentation site for Blizzard Entertainment's Battle.net and in-game network protocols. It replaces a PHP/MariaDB application with pre-generated JSON data files and a vanilla-JS browser frontend.

Production site: https://bnetdocs.org (served from this repo via GitHub Pages)
Source PHP app: https://github.com/BNETDocs/bnetdocs-web

## Architecture

### Two-Stage Build

**Stage 1 — Data export** (run manually from a machine with DB access):
- `bin/export.php` in the `BNETDocs/bnetdocs-web` PHP repo connects to the production MariaDB database, pre-renders all Markdown to HTML using Parsedown, and writes the JSON files in `data/` in this repo.
- The operator commits and pushes the updated `data/` files to trigger a rebuild.

**Stage 2 — Static build** (GitHub Actions, triggered by any push to `main`):
1. `npm run build` — `scripts/build.js` reads `data/` JSON files, generates one `index.html` shell per item, generates index pages, and copies all assets to `dist/`.
2. `npm run pagefind` — Pagefind crawls `dist/` and writes a search index to `dist/pagefind/`.
3. The `dist/` artifact is deployed to GitHub Pages.

### Directory Layout

```
data/               # JSON source of truth — committed, generated by export script
  packets/          # index.json + {id}.json per packet
  documents/        # index.json + {id}.json per document
  news/             # index.json + {id}.json per news post
  products.json     # Blizzard product lookup table
  application-layers.json
  transport-layers.json
  news-categories.json
scripts/
  build.js          # Node.js build script — reads data/, writes dist/
src/
  js/               # Browser JavaScript
    app.js          # Entry point — reads window.BNETDOCS_PAGE, dispatches to renderer
    renderers/      # packet.js, document.js, news.js, search.js, etc.
  css/
    style.css       # Custom styles (extends Bootstrap 4 Slate)
  templates/        # HTML template strings used by build.js
  public/           # Static assets copied verbatim into dist/assets/
dist/               # Build output (gitignored — do not commit)
```

### URL Structure

URLs mirror the original PHP site exactly to preserve existing inbound links:
- `/packet/123/sid-null/` — packet detail (index.html shell, JS hydrates from `/packet/123.json`)
- `/packet/123/` — JS redirect to canonical slug URL
- `/packet/index.json` — JSON API (copy of `data/packets/index.json`)
- `/document/456/title-slug/` — document detail
- `/news/1/slug/` — news post detail
- `/news.rss` — pre-generated RSS feed
- `404.html` — handles legacy URL patterns and slug mismatches

### JSON Data Format

Each index file contains a `generated_at` ISO 8601 timestamp and an array of summary objects. Detail files contain the full object including pre-rendered HTML and comments. Author attribution uses `author_username` (string) only — no user IDs or PII.

### Frontend Design

- Dark theme: `#0d1117` background, `#161b22` cards, `#00ccad` teal accent, `#58a6ff` links
- Bootstrap 4.4.1 (Slate theme) as layout base
- Vanilla JS with `fetch()` — no framework, no build tool for the frontend
- Pagefind for client-side full-text search
- Highlight.js for syntax highlighting in packet format blocks
- JetBrains Mono (self-hosted WOFF2) for monospace elements

## Commands

```bash
npm install              # Install dependencies (pagefind)
npm run build            # Generate dist/ from data/ and src/
npm run pagefind         # Build search index into dist/pagefind/
```

## What Is Cut vs. the PHP App

The following PHP app features are intentionally absent in this static version:
- User accounts (login, register, profiles) — removed entirely
- Comment submission — existing comments shown as read-only archive; Discord link for discussion
- Server status monitoring — removed entirely
- Admin dashboard and event log — removed
- `/search.json` endpoint — returns 404 (use Pagefind instead)
- Dynamic code generation views (C++/Go/Java/PHP/VB) — pre-generated as static files
